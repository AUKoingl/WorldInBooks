## 1. 概述

### 1.1 产品定位

**书中界 (ShuZhongJie)** 是一款面向小说创作者的辅助工具，帮助作者管理小说中的核心要素：**人物、事件、地点、世界观**，并提供它们之间的关联关系可视化和一致性检查功能。

### 1.2 目标用户

- **主要**: 网络小说作者、传统文学作家
- **次要**: 编剧、游戏世界观设计师、TRPG 玩家

### 1.3 产品愿景

成为小说创作者的首选工具，让创作更有条理，让设定更加严谨。

## 2. 核心功能

### 2.1 四维核心系统

| 系统 | 功能 | 核心卖点 |
|------|------|----------|
| 🌍 世界观系统 | 世界背景、规则体系、势力关系、时间线/历史 | 完整的世界构建框架 |
| 👥 人物系统 | 角色档案、关系网络、成长弧光、动机追踪 | 立体的人物塑造工具 |
| 📖 事件系统 | 事件因果链、情节管理、伏笔追踪、冲突设计 | 情节逻辑自动梳理 |
| 📍 地点系统 | 场景描述、空间关系、地点 - 事件/人物关联 | 可视化的地理管理 |

### 2.2 差异化功能

#### 2.2.1 关系图谱（核心卖点）
- **四维关联可视化**: 人物↔事件地点↔世界观 的双向关联
- **力导向图**: 交互式探索复杂关系网络
- **路径分析**: 找出两个角色之间的隐藏联系
- **影响力计算**: 量化角色在故事中的重要程度

#### 2.2.2 时间轴系统
- **多视图**: 水平/垂直时间线，支持缩放
- **人物轨迹叠加**: 同时查看多个人物的时间线
- **冲突检测**: 自动发现时间线矛盾（如：同一人物同时出现在两地）

#### 2.2.3 因果链追踪
- **事件树**: 可视化事件的起因和结果
- **伏笔管理**: 标记伏笔及其回收位置
- **环路检测**: 发现因果循环（A 导致 B，B 导致 A）

#### 2.2.4 一致性检查（核心卖点）
- **时间线冲突**: 自动检测时间矛盾
- **人物设定矛盾**: 发现前后不一致的人物特征
- **地点逻辑验证**: 检查距离与时间的合理性
- **事件因果审查**: 验证因果链的完整性

### 3.2 技术栈选择

| 层级 | 技术 | 理由 |
|------|------|------|
| **应用框架** | Tauri v2 | 比 Electron 更小（~10MB vs ~150MB），Rust 后端性能优异 |
| **前端** | React 18 + TypeScript | 生态成熟，类型安全，便于协作 |
| **UI 库** | TailwindCSS + shadcn/ui | 高性能、可定制、现代化设计 |
| **状态管理** | Zustand | 轻量（~1KB）、简单、性能优于 Redux |
| **本地数据库** | SQLite (tauri-plugin-sql) | 零配置、单文件、足够快、ACID 事务 |
| **图数据库** | Neo4j (本地嵌入/可选) | 处理复杂关系查询的最佳选择 |
| **图谱可视化** | Cytoscape.js | 功能强大、支持 WebGL 加速 |
| **数据验证** | Zod | TypeScript 优先的 schema 验证 |
| **构建工具** | pnpm + Turborepo | 依赖共享、增量构建、并行执行 |

### 3.3 数据流向（单向数据流）

```
┌──────────┐     ┌──────────┐     ┌──────────     ┌──────────────┐     ┌──────────────┐
│ UI 组件   │ ──► │ 状态管理  │ ──► │ 服务层    │ ──► │ Repository   │ ──► │ SQLite/Neo4j │
│ 用户交互  │     │ Zustand  │     │ 业务逻辑  │     │ 数据抽象     │     │ 持久化       │
└──────────┘     └──────────┘     └──────────     └──────────────     └──────────────
```

## 5. 性能目标

| 指标 | 目标值 | 实现策略 |
|------|--------|----------|
| UI 响应时间 | < 100ms | 虚拟滚动、懒加载、Memoization |
| 查询延迟 | < 1s | 内存缓存、数据库索引优化 |

| 图谱渲染 | 增量 | 按需加载、LOD 技术 |

| 安装包大小 | < 30MB | Tauri 精简、资源压缩 |

## 6. 模块化开发策略

### 6.1 模块划分原则

- **高内聚**: 相关代码放在一起
- **低耦合**: 模块之间通过明确接口通信
- **单一职责**: 每个模块只做一件事

### 6.2 可并行开发的模块

```
核心层（必须顺序完成）
├── entities        ──► 基础，最先完成
└── repository      ──► 依赖 entities

功能层/服务层/UI 层（核心层完成后可并行）
├── 人物模块 (独立开发者 A)
├── 地点模块 (独立开发者 B)
├── 事件模块 (独立开发者 C)
├── 世界观模块 (独立开发者 D)
├── 图谱可视化 (独立开发者 E)
└── 时间轴组件 (独立开发者 F)
```

### 6.3 模块间通信

所有模块通过 Repository 层定义的接口进行数据访问，不直接操作数据库：

```typescript
// 正确：通过 Repository 访问
const characters = await characterRepository.findAll();

// 错误：直接操作数据库
const characters = await db.select('*').from('characters');
```

## 7. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Neo4j 学习曲线 | 开发延期 | 提前准备 Neo4j 教程，先实现 SQLite 版本 |
| 图谱性能问题 | 用户体验差 | 使用 WebGL 加速，实现 LOD 和增量渲染 |
| Tauri v2 不成熟 | 遇到 Bug | 关注官方 Discord，准备 fallback 方案 |
| 模块化过度设计 | 开发效率低 | YAGNI 原则，每阶段只实现必要功能 |
